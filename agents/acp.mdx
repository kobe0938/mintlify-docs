Harbor includes a built-in runner for agents published in the [Agent Client Protocol (ACP) registry](https://github.com/agentclientprotocol/registry). Instead of writing a custom integration, you reference an agent by its registry ID and optional version, and Harbor fetches the agent definition, installs it into the container, and runs it — all automatically. This gives you immediate access to a growing ecosystem of community-published agents without any bespoke glue code.

## Running an ACP Agent from the CLI

Use the `acp:<id>[@version]` shorthand anywhere `--agent` is accepted. The following example runs `opencode` at a pinned version:

```bash
harbor run \
  --path examples/tasks/hello-world \
  --agent acp:opencode@1.3.9 \
  --model openai/gpt-5.4 \
  --ae OPENAI_API_KEY=$OPENAI_API_KEY
```

If you omit the version, Harbor resolves the latest `agent.json` from the registry's `main` branch during agent setup:

```bash
harbor run \
  --path examples/tasks/hello-world \
  --agent acp:opencode \
  --model openai/gpt-5.4 \
  --ae OPENAI_API_KEY=$OPENAI_API_KEY
```

<Note>
  Version resolution for unpinned agents requires a network request at setup time. Pin the version in production runs to ensure reproducibility.
</Note>

## Using ACP Agents in Config Files

The same `acp:<id>[@version]` shorthand works in YAML or JSON config files, and Harbor preserves it in the persisted `config.json`:

```yaml
agents:
  - name: acp:opencode@1.3.9
    model_name: openai/gpt-5.4
    kwargs:
      auth_policy: auto
      permission_mode: allow
```

## Using ACP Agents with the SDK

SDK users pass the agent name as a plain string in `AgentConfig`. Registry resolution happens asynchronously inside `AcpAgent.setup()`, so constructing the config object performs no network I/O:

```python
from harbor.models.trial.config import AgentConfig

agent = AgentConfig(
    name="acp:opencode@1.3.9",
    model_name="openai/gpt-5.4",
    env={"OPENAI_API_KEY": "${OPENAI_API_KEY}"},
)
```

## Configuration Options

Pass ACP-specific options through `kwargs` in your config:

| Option | Description |
|---|---|
| `auth_policy` | How to handle authentication: `auto`, `explicit`, or `disabled` |
| `permission_mode` | Whether to allow or deny agent permissions: `allow` or `deny` |
| `distribution_preference` | Comma-separated preference order among `binary`, `npx`, and `uvx` |
| `registry_ref` | Git ref to use when resolving the latest version from the registry |
| `registry_cache_dir` | Local directory for caching fetched registry entries |

You can also skip registry resolution entirely and point Harbor at a local agent definition file:

```yaml
agents:
  - name: acp
    kwargs:
      registry_entry_path: /path/to/agent.json
```

## Output Files

Every ACP run writes the following files under the agent log directory:

| File | Contents |
|---|---|
| `acp.txt` | Human-readable agent output log |
| `acp-events.jsonl` | Raw ACP event stream, one JSON object per line |
| `acp-summary.json` | High-level run summary |
| `trajectory.json` | ATIF-format trajectory generated by Harbor |

Because Harbor generates `trajectory.json` in standard [ATIF format](/agents/trajectory-format), you can export traces the same way you would for any other agent:

```bash
harbor traces export -p jobs/<job-name> --recursive
```
